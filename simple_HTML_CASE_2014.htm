3             Introduction

3.1         Linked Data

Whether in government, companies or at school, data is constantly present and permeates our lives more and more. Often times we have the problem to make sense out of this big data flood, take full advantage of it and make it valuable. To make data valuable we have to enrich it with some sense and build relationships among different data sets. If we enhance the data by linking it with other sources, better decisions can be made, predications on outcomes are more precise and we get a better understanding on how things actually work[RS3] .

 

This is where Linked Data comes in. Linked Data is part of the Semantic Web and describes free available structured data, which is reachable through Uniform Resource Identifiers (URI). It is published in a format (often RDF) so that it can be interlinked with other data, and through that be more useful. Figure XY[RS4]  shows the concept of linked data in an abstract way[RS5] .

3.2         Workflows

Before we use actual data, we have to ask ourselves one question: Can we trust the data?

This question is hard to answer with just the data itself. We have to know where the data comes from, when it was raised, how it was processed and so forth.

 

Workflows make it possible to recreate the whole procedure on how the data got created. We can assign attributes to the workflows to indicate who went through the workflow, when and where it was run and so on.

 

Workflows have three major benefits:

-       Provenance: It strengthens trust in the result of the workflow.

-       Reproducibility: The reproducibility process can be very cost intensive. Therefore a workflow serves other scientists to re-execute them and reproduce their results in a cost saving way.

-       Validation: It validates the method that led to a new scientific insight[RS6] .

3.2.1      Workflows for Linked Data

In the case of Linked Data we have to go through a form of the ETL-process. The ETL-process is mostly known and has its usage in Data Warehousing (http://de.wikipedia.org/wiki/ETL-Prozess). [RS7] ItÕs a process, where data from different structured data sources get integrated into a single data store. First the data has to get extracted from the different sources and then mapped into a suitable structure for the target data store and finally loaded into the store.

The ETL-process for Linked Data contains the following steps:

-       Inspection
In this step questions like the following have to be answered: Who is the publisher of the data? How big is the data to be extracted? What is the licence behind the data?

-       Extraction
The extraction step contains tasks that have to be executed to get a local data dump.

-       Pre-processing
In this workflow step the data dump has to get ready to be maped. Also, the necessary tooling has to be in order to work with the data dump.

-       Mapping
In the mapping step the data dump gets transformed into a RDF format, so it can finally be loaded into the target data store.

-       Loading
In this step the mapped data gets loaded into the target store. The last step is not in the scope of this CASE.

3.4         Goal for this CASE

The Goal of this CASE is to design workflows for the ETL-process in a Linked Data environment. The focus lies on the workflow steps Inspection, Extraction, Pre-processing and Mapping. The step Loading is not in the scope of this CASE. The workflows are tested and deployed for the 270a.info Linked Dataspace for data sets from Transparency International and Doing Business.

The workflows will get enhanced with additional RDF, so they can be published as linked data and easily reproduced.

 

 

4             ETL for Transparency International

Transparency International is a none-profit-organization, which is engaged into fighting corporate and political corruption. The 270a.info data space already contains data for the Corruption Perceptions Index (CPI) for the years 2009 – 2011 calculated by Transparency International. The CPI is an index that ranks countries/territories based on how corrupt a countryÕs public sector is perceived to be. It is the most widely used indicator of corruption worldwide.

Chapter 4 describes how the workflows for the CPI 2012 and 2013 were created.

4.1         Drake

To manage and execute the created workflows a tool called drake was used. Drake is a text-based data workflow tool, which is able to create workflow steps and organize their execution. It works similar to GNU Make. Drake uses an output-input-approach to create dependencies between the different workflow steps. The tool is free for use and available on github[RS9] .

4.2         Traql

Tarql is a tool that is used in the mapping step of the workflow. It is a java based command-line tool for converting CSV files to RDF by using SPARQL 1.1 syntax. Tarql is free for use and available on github (https://github.com/cygri/tarql).

Another important discovery of the inspection is the use of different country codes in the Excel files from Transparency International compared to the URI pattern that is used on the 270a.info data space.

Figure 4: Country codes Excel files

Figure 3 shows that Transparency International uses other country codes than Sarven Carpadisli does in his URI patterns on 270a.info:

4.4         Extraction

The extraction is a fairly easy workflow step. Two things have to be done:

1.      Extract the CPI data from the website of Transparency International.

2.      Extract the country codes from the 270a.info dataspace.

These tasks are done by the simple bash command wget (see also.. [RS11] ).

4.5         Pre-processing

The raw data is now available as a local dump. In this workflow step the data has to be prepared for the mapping with Tarql (see chapter 5.2 Tarql). Tarqel assumes that the CSV file has column names in the first row, and uses those column names as variable names. Because of that, the Excel file with the CPI data has to be converted to CSV and refined to match this requirement.

The end result of the Pre-processing step is a CSV file that looks like Figure 4.

Figure 5: Refined CPI data

As shown in Figure 4, the country codes downloaded in the extraction step got merged with the CPI data. This was done by the bash command join (see also[RS12] ).

This workflow step is responsible for mapping the merged CSV file to RDF (turtle) by using Tarql (see also[RS13] ).

4.6.1      Mapping problem

After mapping the CSV file to RDF, a problem occurred: Tarql mapped also empty cells from the CSV file. The following example illustrates the problem:

Denmark,DK,1,90,7,2,87,93,83,98,,89,,95,98,,86,89,88,83,,,

The last 13 values of this row (including the empty ones) represent sources (surveys) and their results, which are used to calculate the CPI for Denmark. Empty values mean, that there was no survey in this case. The final RDF representing the observation for Denmark should not include empty values.
